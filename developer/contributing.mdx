<Note>
  See [Building from Source](/developer/building) for detailed setup instructions.

### Quick Start

```bash
# Clone the repository
git clone https://github.com/ModioStudio/MiniFy.git
cd MiniFy

# Install dependencies
pnpm install

# (Optional) Install git hooks
pnpm dlx lefthook install

# Start development
pnpm desktop:dev
```

### Prerequisites

- Node.js 18+
- pnpm 9+
- Rust toolchain (latest stable)
- Platform-specific Tauri dependencies

## Contribution Workflow

<Steps>
  <Step title="Fork and Clone">
    Fork the repository on GitHub, then clone your fork:
    
    ```bash
    git clone https://github.com/YOUR_USERNAME/MiniFy.git
    cd MiniFy
    git remote add upstream https://github.com/ModioStudio/MiniFy.git
    ```
  </Step>
  
  <Step title="Create a Branch">
    Create a descriptive branch for your changes:
    
    ```bash
    git checkout -b fix/spotify-polling-rate
    # or
    git checkout -b feat/new-player-layout
    ```
    
    **Branch naming conventions:**
    - `fix/` - Bug fixes
    - `feat/` - New features
    - `docs/` - Documentation changes
    - `refactor/` - Code refactoring
    - `test/` - Test additions/changes
  </Step>
  
  <Step title="Make Changes">
    Write your code following the project's style and conventions.
    
    <Note>
      Run `pnpm lint` frequently to catch issues early.
    </Note>
  </Step>
  
  <Step title="Test Your Changes">
    Test thoroughly in development mode:
    
    ```bash
    # Desktop app
    pnpm desktop:dev
    
    # Run linter
    pnpm lint
    
    # Format code
    pnpm format
    ```
    
    <Warning>
      Always test production builds before submitting:
      ```bash
      pnpm desktop:build
      ```
    </Warning>
  </Step>
  
  <Step title="Commit Changes">
    Write clear, concise commit messages:
    
    ```bash
    git add .
    git commit -m "fix: resolve polling rate issue when player is paused"
    ```
    
    **Commit message format:**
    - `fix: <description>` - Bug fix
    - `feat: <description>` - New feature
    - `docs: <description>` - Documentation
    - `refactor: <description>` - Code refactoring
    - `style: <description>` - Code style/formatting
    - `test: <description>` - Tests
    - `chore: <description>` - Build/tooling
  </Step>
  
  <Step title="Push and Create PR">
    Push your branch and create a pull request:
    
    ```bash
    git push origin fix/spotify-polling-rate
    ```
    
    Then open a PR on GitHub with:
    - Clear title describing the change
    - Description explaining the problem and solution
    - Steps to test the changes
    - Link to related issues
  </Step>
</Steps>

## Code Style Guidelines

### TypeScript/React

<CardGroup cols={2}>
  <Card title="Biome" icon="broom">
    We use Biome for linting and formatting (replaces ESLint + Prettier)
    
    ```bash
    pnpm check
    pnpm format
    ```
  </Card>
  
  <Card title="TypeScript" icon="code">
    Always use TypeScript. Avoid `any` types.
    
    ```typescript
    // Good
    const track: SpotifyTrack = {...}
    
    // Avoid
    const track: any = {...}
    ```
  </Card>
</CardGroup>

### Component Structure

```typescript
// components/PlayerControls.tsx
import { usePlaybackStore } from '@/stores/playback';
import { Play, Pause, SkipForward } from '@phosphor-icons/react';

export function PlayerControls() {
  const { isPlaying, play, pause, next } = usePlaybackStore();
  
  return (
    <div className="flex gap-2">
      <button onClick={isPlaying ? pause : play}>
        {isPlaying ? <Pause /> : <Play />}
      </button>
      <button onClick={next}>
        <SkipForward />
      </button>
    </div>
  );
}
```

### Rust Code

<CardGroup cols={2}>

### Tauri Commands

```rust
// src-tauri/src/commands/spotify.rs
use tauri::State;
use crate::spotify::SpotifyClient;

#[tauri::command]
pub async fn get_playback(
    client: State<'_, SpotifyClient>
) -> Result<PlaybackState, String> {
    client
        .get_current_playback()
        .await
        .map_err(|e| e.to_string())
}
```

## Pull Request Guidelines

### What Makes a Good PR

<Steps>
  <Step title="Scope Small">
    Focus on solving one problem. Avoid combining unrelated changes.
    
    <Check>**Good**: Fix Spotify token refresh logic</Check>
    <X>**Avoid**: Fix token refresh + add new theme + update dependencies</X>
  </Step>
  
  <Step title="Write Clear Descriptions">
    Explain what the PR does and why:
    
    ```markdown
    ## Problem
    When the Spotify access token expires, the app doesn't automatically 
    refresh it, causing playback controls to fail.
    
    ## Solution
    Added automatic token refresh logic that checks expiration before 
    each API call and requests a new token using the refresh token.
    
    ## Testing
    1. Wait for token to expire (1 hour)
    2. Try to play/pause
    3. Verify playback controls still work
    ```
  </Step>
  
  <Step title="Link Related Issues">
    Reference GitHub issues in your PR description:
    
    ```markdown
    Fixes #123
    Related to #456
    ```
  </Step>
  
  <Step title="Update Documentation">
    If your change affects user-facing behavior or developer workflows, update relevant documentation.
  </Step>
</Steps>

### PR Checklist

<Accordion title="Before Submitting">
  - [ ] Code follows project style guidelines
  - [ ] Ran `pnpm lint` and fixed all issues
  - [ ] Ran `pnpm format` to format code
  - [ ] Tested in development mode (`pnpm desktop:dev`)
  - [ ] Tested production build (`pnpm desktop:build`)
  - [ ] Updated documentation if needed
  - [ ] Linked related GitHub issues
  - [ ] PR description explains the change clearly
</Accordion>

## Testing

### Manual Testing

<Steps>
  <Step title="Development Testing">
    Test your changes in development mode:
    
    ```bash
    pnpm desktop:dev
    ```
    
    Use DevTools to debug:
    - **macOS**: `Cmd + Option + I`
    - **Windows/Linux**: `Ctrl + Shift + I`
  </Step>
  
  <Step title="Production Testing">
    Always test production builds:
    
    ```bash
    pnpm desktop:build
    ```
    
    Install and run the built application from:
    - **macOS**: `apps/desktop/src-tauri/target/release/bundle/macos/MiniFy.app`
    - **Windows**: `apps/desktop/src-tauri/target/release/bundle/nsis/`
    - **Linux**: `apps/desktop/src-tauri/target/release/bundle/appimage/`
  </Step>
  
  <Step title="Cross-Platform Testing">
    If possible, test on multiple platforms:
    - macOS (Intel + Apple Silicon)
    - Windows 10/11
    - Linux (Ubuntu, Arch, Fedora)
  </Step>
</Steps>

### Reporting Issues

When reporting bugs, include:

<Steps>
  <Step title="Environment">
    - OS and version (e.g., macOS 14.2, Windows 11, Ubuntu 22.04)
    - MiniFy version
    - Node.js and Rust versions
  </Step>
  
  <Step title="Steps to Reproduce">
    Clear, numbered steps to reproduce the issue:
    
    1. Launch MiniFy
    2. Click play button
    3. Observe error in console
  </Step>
  
  <Step title="Expected vs Actual">
    - **Expected**: Playback should start
    - **Actual**: Error: "Token expired"
  </Step>
  
  <Step title="Logs/Screenshots">
    Include relevant logs, error messages, or screenshots.
    
    <Note>
      Open DevTools to view console logs in development mode.
    </Note>
  </Step>
</Steps>

## Code of Conduct

### Be Respectful

- Be concise and respectful in all communications
- Focus on technical merit, not personal preferences
- Accept constructive feedback gracefully
- Help others learn and grow

### Stay On Topic

- Keep discussions focused on the issue at hand
- If stuck, describe what you tried and where you're blocked
- Link to relevant code, docs, or issues when helpful
